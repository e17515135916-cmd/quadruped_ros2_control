# Gazebo 图形界面问题分析

## 问题描述
Gazebo 客户端（gzclient）启动后卡在 logo 界面，无法显示 3D 场景。

## 根本原因
gzclient 在加载机器人的 STL mesh 文件时卡住。可能的原因：

1. **Mesh 文件过大或过于复杂** - STL mesh 文件包含大量三角形，加载和渲染需要时间
2. **渲染引擎问题** - OGRE 渲染引擎在处理复杂 mesh 时可能遇到性能问题
3. **OpenGL 驱动问题** - 系统的 OpenGL 驱动可能不完全支持 Gazebo 所需的功能

## 已确认的事实
- ✅ gzserver（物理引擎）正常运行
- ✅ 机器人成功 spawn 到仿真中
- ✅ 物理仿真正在计算（机器人位置在变化）
- ❌ gzclient 卡在 logo 界面
- ❌ 机器人掉到地面以下（Z ≈ -1.0）

## 解决方案

### 方案 1：修复 URDF 碰撞体（推荐）
将复杂的 STL mesh 碰撞体替换为简单的几何原语（cylinder、box、sphere）：
- 减少碰撞检测计算量
- 提高物理仿真稳定性
- 可能解决 mesh 加载问题

### 方案 2：简化 Visual Mesh
如果碰撞体修复后图形界面仍有问题，可以：
- 使用简化的 mesh 文件用于可视化
- 或者使用简单几何体代替 mesh

### 方案 3：使用 RViz 可视化
不依赖 Gazebo 图形界面，使用 RViz 查看机器人状态：
- RViz 可以显示碰撞几何体
- 可以验证 URDF 配置是否正确
- 不需要 Gazebo 图形界面

### 方案 4：升级到 Gazebo Ignition/Harmonic
新版本的 Gazebo 使用不同的渲染引擎，可能解决兼容性问题。

## 当前策略
1. 先修复 URDF 碰撞体配置（按照规范执行任务 4 和 5）
2. 使用命令行工具监控物理仿真
3. 验证机器人不再"掉下去"
4. 如果需要可视化，使用 RViz
5. 最后再解决 Gazebo 图形界面问题

## 临时解决方法
如果急需查看 Gazebo 场景，可以：
1. 等待更长时间（5-10 分钟）让 mesh 加载完成
2. 使用更简单的测试模型验证 Gazebo 是否正常工作
3. 检查系统资源（CPU、内存、GPU）是否充足
